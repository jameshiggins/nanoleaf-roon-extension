# Visualizations

Every visual is rendered here, from the **audio itself** — the PCM stream tapped out of Roon —
and streamed to the panels frame-by-frame over extControl v2 UDP. Nothing uses the Nanoleaf
microphone. On each Roon track change the extension rotates to a fresh **visualizer × palette**
combination, so the look changes with the music while always reacting to the live signal.

```
Roon PCM ─▶ feature extraction ─▶ visualizer.render(features) ─▶ extControl UDP ─▶ panels
            (bands, onsets, levels)   (one of 30+, per-panel)
   Roon track change ───────────────▶ rotate visualizer + palette
```

## What the visuals react to

A lightweight DSP stage (`src/dsp/features.js`, no FFT — cheap one-pole filters) produces, per
frame:

- **level / peak** and **left / right** loudness envelopes,
- **bass / mid / treble** band energies (≈ <250 Hz, 250 Hz–2.5 kHz, >2.5 kHz),
- **energy**, a perceptual blend, and
- **onset** — a beat flag from bass-flux detection with a refractory window.

Visualizers use these to drive brightness, motion speed, spawn events, and hue. A master
**silence gate** fades everything to black when the feed goes quiet, so gaps between tracks look
intentional.

## Visualizer families (30+ types)

Generated by crossing a dozen parametric engines with their variant grids (`npm run visuals`
lists them all with a one-line description):

| Family | Variants | What it does |
| --- | --- | --- |
| **pulse** | stereo, mono, blob | loudness pulse; beats flash; blob grows from the center with the bass |
| **bars** | horizontal, vertical, center | bass/mid/treble columns with peak-hold |
| **ripple** | random, core, implode, twin | beats spawn expanding (or imploding, or mirrored) rings |
| **streaks** | alternate, duel, rain | beat-launched comets across the panels; rain falls in columns |
| **wheel** | spin, counter, rainbow, strobe | hue wheel spinning with the energy; beats flash/strobe |
| **wave** | sweep, fall, cross | a bright ridge sweeping an axis, widened by the bass |
| **sparkle** | glitter, burst | treble-driven glitter over a bass wash; burst ignites clusters on beats |
| **chase** | on-beat, flow | a lit window stepping on beats or flowing with the energy |
| **vu** | meters, tower | classic L/R meters from the center out, or one bottom-up bar |
| **bounce** | single, duo | a block bouncing side to side, sized by the bass |
| **fire** | up, fall | bass-driven flames |
| **sections** | — | beats light random contiguous panel runs |

## Palettes (procedural, 36+)

Palettes are generated, not hand-picked (`src/visuals/palettes.js`): base hues step around the
color wheel by the golden angle (so consecutive palettes are far apart), crossed with six
color-harmony schemes (analogous "Drift", complementary "Clash", triadic "Triad", split, square,
monochrome-pop). Each palette is three hues — a background wash, the moving elements, and the
beat hits. The default is 36; raise `visuals.palettes` for more (they stay distinct into the
hundreds). 30+ visualizers × 36 palettes ≈ **1,000+ distinct looks** in rotation.

## Configuration

```json
{
  "nanoleaf": { "fps": 30 },
  "visuals": {
    "include": [],          // limit rotation to these visualizers; empty = all
    "exclude": [],          // skip specific visualizers
    "palettes": 36,         // how many palettes to generate (>= 1)
    "rotate": "track",      // "track" | "off" | <seconds>
    "minSeconds": 8,        // don't rotate faster than this on rapid track skipping
    "gain": 1.0,            // boost quiet sources so visuals reach full swing
    "attackMs": 5,          // envelope rise (snappier beats = lower)
    "releaseMs": 180,       // envelope fall (longer trails = higher)
    "silenceFloor": 0.02    // energy below this fades panels to black
  }
}
```

- **`rotate`**: `"track"` switches on every Roon track change (needs `roon.enabled`), `"off"`
  stays on one look, or a number rotates every N seconds regardless of Roon.
- **`include` / `exclude`**: names from `npm run visuals`, case-insensitive. Pin a favorite with
  `"include": ["ripple"]`, or drop ones you dislike with `exclude`.
- Rotation is a **shuffle bag**: every visualizer (and palette) is used once before any repeats,
  and never the same one twice in a row.

## Checking a track really goes 0-to-max

To confirm the visualization uses the full range on real music — quiet parts near black,
loud parts at full brightness — run a song through the actual analysis + render pipeline offline:

```bash
npm run analyze -- /path/to/song.flac                 # needs ffmpeg on PATH
npm run analyze -- song.mp3 --gain 1.8 --visual bars  # try a gain / a specific visual
node tools/analyze.js capture.raw --raw               # s16le 44100 stereo, no ffmpeg
```

It prints the dynamic range of every audio feature (rms, bass/mid/treble, energy), the resulting
**on-panel brightness** (peak % of full, median, a timeline sparkline across the track), and a
verdict:

```
On-panel brightness (0-255):
  peak     █████████████··············· 119  (47% of full)
  ...
Verdict:
  ✗ Under-driven — peaks only reach 47% of full. Set visuals.gain to ~1.68 so loud parts hit max.
  ✓ Quiet passages fade toward black — good 0-to-max contrast across the track.
```

If peaks fall short of full, it tells you the `visuals.gain` to set so the loudest moments hit
maximum brightness. This is the real DSP + renderer, so what it measures is what the panels do.

## Tuning the feel

- Visuals too dim on quiet recordings → raise `visuals.gain` (e.g. `1.5`–`2.5`).
- Beats feel mushy → lower `attackMs` and/or `releaseMs`.
- Panels flicker to black between beats → raise `releaseMs`, or `silenceFloor` if it's noise.
- Want a specific vibe → `"include"` a subset (e.g. only `["wheel-rainbow","ripple-core","wave-cross"]`).
